## 3. 集成边界

### 3.1 与现有代码的关系

当前仓库为单一 `index.html` 文件的纯静态应用（2048 游戏），无前端框架、无构建流程、无路由。营销页的集成存在以下边界选择：

| 方案 | 描述 | 优劣 |
|------|------|------|
| **A. 独立 HTML 文件** | 在根目录新增 `marketing.html`（或 `promo/index.html`），与 `index.html` 并列 | 最低侵入性，独立部署，不影响现有游戏页面 |
| **B. 子目录隔离** | 新建 `marketing/` 目录，内含完整页面资源 | 资源隔离清晰，便于后续扩展为多页应用 |
| **C. 引入构建工具** | 添加 Vite/Webpack，将全部页面纳入构建管线 | 适合长期发展，但初始成本高，改变现有部署流程 |

**建议**: 采用方案 B（子目录隔离），在 `marketing/index.html` 中实现营销页。理由：
- 不修改现有 `index.html`
- 资源（图片、CSS、JS）可在 `marketing/` 目录内自包含
- 服务器路由只需按目录映射，无需构建步骤
- 后续如需引入构建工具可平滑过渡

### 3.2 技术栈建议

| 维度 | 建议 | 理由 |
|------|------|------|
| HTML | 语义化 HTML5 | 保持与现有代码一致 |
| CSS | 独立 `<style>` 或外联 CSS 文件 | 移动端优先，rem/vw 适配 |
| JS | 原生 JS 或轻量库 | 仓库无构建工具，避免引入编译依赖 |
| 移动适配 | `<meta name="viewport">` + rem/vw | H5 营销页核心场景为移动端 |
| 图片 | WebP + PNG fallback，懒加载 | 减少首屏加载时间 |
| 分享 | 微信 JS-SDK（按需） | 需后端提供签名接口 |

### 3.3 页面路由位置

```
/                       → index.html（2048 游戏，保持不变）
/marketing/             → marketing/index.html（营销页）
/marketing/assets/      → 营销页静态资源（图片、字体等）
```

## 4. 实现约束清单

以下约束在后续开发中必须遵循：

### 4.1 硬约束（不可违反）

1. **不修改 `index.html`** — 营销页不得改动现有 2048 游戏代码
2. **纯静态交付** — 在引入构建工具之前，营销页必须可作为静态文件直接服务
3. **移动优先** — 页面必须在 320px–428px 宽度范围内正常显示，目标设备为手机浏览器/微信内置浏览器
4. **无构建依赖** — 不引入 npm/yarn、Webpack/Vite 等构建步骤（除非后续任务明确要求）
5. **中文环境** — `lang="zh-CN"`，UTF-8 编码，中文标点规范
6. **安全合规** — 用户输入（如表单）必须做前端校验；不在前端暴露 API 密钥；遵循个人信息保护规范

### 4.2 软约束（强烈建议）

1. **首屏加载 ≤ 3 秒（3G 网络）** — 控制首屏资源总量在 500 KB 以内
2. **图片优化** — 使用 WebP 格式，配合 `<picture>` 标签提供 PNG/JPG 降级
3. **分享元信息** — 配置 Open Graph 和微信分享 meta 标签
4. **CSS 隔离** — 营销页样式不影响其他页面（通过目录隔离或命名空间实现）
5. **埋点预留** — 为关键交互（CTA 点击、页面曝光、分享成功）预留数据埋点接口
6. **可配置性** — 文案、链接、图片路径等运营参数集中管理，便于复用模板
